# Security & Quality Audit — 2026-03-26

Findings from senior code review. CRITICAL and HIGH items were fixed in the
same session. Items below are deferred for later.

## Fixed (this session)

- **[CRITICAL C2]** `RotateKey` now copies `newKey` before passing to C and
  zeroes the copy on return. Prevents caller mutation during the C call.
- **[HIGH H3]** `rows.Close()` now acquires `conn.mu` before `sqlite3_finalize`.
- **[HIGH H4]** `stmt.Close()` now acquires `conn.mu` before `sqlite3_finalize`.
- **[HIGH H1]** Removed `newConnector()` — was inconsistent with `Open()` for
  WAL-on-memory. Test uses `buildConnector()` (extracted from `Open()`).
- **[LOW L1]** Simplified redundant `retErr` in `RotateKey`.
- **[LOW L3]** `TestConnectorKeyZeroedOnClose` now verifies bytes are zeroed,
  not just that the slice is nil.

## Deferred

### C1 — Document GC limitation for key zeroing

`connector.Close()` zeroes the current backing array, but Go's GC may have
copied the array during compaction. Prior copies are unreachable and cannot be
zeroed. This is an inherent Go limitation.

**Action:** Add a doc comment on `connector.Close()` and in the package doc
explaining that key zeroing is best-effort due to GC. Evaluate whether `mlock`
(via syscall) is warranted for the threat model.

### H2 — Zero/update connector key after RotateKey

After successful rekey, the connector still holds the old key. The caller must
close and reopen, but there is no programmatic enforcement. The old key remains
in memory until `db.Close()`.

**Action:** After successful rekey, zero the old key in the connector and
replace it with the new key. This requires `RotateKey` to access the connector
(e.g. via `db.Conn().Raw()` to reach the `conn`, then back to the connector,
or by storing a connector reference accessible from the `*sql.DB`).

### M1 — `runtime.KeepAlive` for blob bind args

In `bindArgs` (`conn.go`), the `[]byte` case passes `unsafe.Pointer(&v[0])` to
`securedb_bind_blob`. Although `SQLITE_TRANSIENT` causes an immediate copy and
`v` is held alive by the `args` slice, an explicit `runtime.KeepAlive(v)` after
the call would be defense-in-depth.

### M2 — Comment why only `:memory:` is checked (not URI paths)

`isMemoryPath` only matches the literal `:memory:`. SQLite also recognizes
`file::memory:` and `file::memory:?cache=shared` when URI filenames are enabled.
Since we do not pass `SQLITE_OPEN_URI`, this is safe, but a code comment should
explain this so a future maintainer who adds URI support knows to update it.

### M3 — Document multi-statement exec behavior difference

`conn.ExecContext` fast path (no args) uses `sqlite3_exec` which executes all
semicolon-separated statements. The slow path (with args) uses
`sqlite3_prepare_v2` which only prepares the first statement. This is a subtle
correctness difference that should be documented.

### M6 — Hardcoded key in `Available()`

`Available()` uses `[]byte("securedb-availability-check-key!")` which is visible
in the compiled binary. Not a real security issue (throwaway temp DB), but could
use a random key like the tests do, or add a comment explaining why it is safe.

### L2 — `BeginTx` ReadOnly uses BEGIN DEFERRED

`BEGIN DEFERRED` does not actually enforce read-only semantics in SQLite. If the
caller expects `ReadOnly: true` to prevent writes, they will be surprised.
Consider documenting this limitation or returning an error when `ReadOnly` is
requested on a read-write connection.

### L4 — Darwin CGO flags assume Apple Silicon path

`doc.go` hardcodes `/opt/homebrew/opt/sqlcipher/` which works for Apple Silicon
but not Intel Macs (`/usr/local/opt/sqlcipher/`). Consider adding both paths or
using `pkg-config`.
